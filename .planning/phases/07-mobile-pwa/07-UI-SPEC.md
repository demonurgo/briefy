# Phase 7 — UI Design Contract: Mobile + PWA

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

This phase is **adaptive, not additive**. It makes the existing UI work on mobile without altering content, data, or navigation structure. The design contract below governs the responsive behavior and interaction details for each component being adapted.

**In scope:** overflow-x fix, Kanban mobile scroll isolation, DemandDetailModal full-screen on mobile, Dashboard single-column on mobile, PWA manifest + icons.

**Out of scope (D-06):** Service worker caching, offline shell, push notifications. MOB-04 ("core pages navigable offline") is partially deferred per CONTEXT.md decision D-06. The sw.js stub already present in `public/sw.js` is a passthrough-only fetch handler — no cache-first strategy is added this phase.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap-1`), badge padding |
| sm | 8px | Compact element spacing (`p-2`, `gap-2`) |
| md | 16px | Default element spacing (`p-4`, `gap-4`) |
| lg | 24px | Section padding (`p-6`, `gap-6`) |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- **Touch targets:** All tappable elements on mobile must be a minimum of **44×44px** (per iOS HIG / WCAG 2.5.5). Existing BottomNav items are already `h-16` (64px) — compliant. Bell icon button is `h-8 w-8` (32px) — it must receive `min-h-[44px] min-w-[44px]` on mobile via responsive class addition.
- **BottomNav height:** Fixed at `h-16` (64px). The `main` element already has `pb-20 md:pb-6` to clear it — maintain.
- **Kanban column width:** Fixed at `w-72` (288px) per column — this is intentional per D-01 (horizontal scroll stays, not stacked).
- **Modal full-screen on mobile:** `inset-0` (`top: 0; right: 0; bottom: 0; left: 0`) with `p-0` replacing `p-4` at breakpoint below `md:`.

## Responsive Breakpoints

**Primary breakpoint:** `md:` (768px) — established by BottomNav.tsx using `md:hidden`. All Phase 7 responsive work uses this single breakpoint consistently.

Do not introduce `sm:` as a primary mobile breakpoint for new mobile-specific rules. `sm:` (640px) is only used where it already exists in Dashboard grids.

| Breakpoint | Behavior |
|------------|----------|
| below `md:` (<768px) | Mobile layout: BottomNav visible, sidebar hidden, full-screen modal, single-column dashboard, Kanban horizontal scroll |
| `md:` and above (≥768px) | Desktop layout: sidebar visible, BottomNav hidden, modal at max-w-6xl centered, Kanban columns side-by-side |

## Component Contracts

### 1. AppLayout — Overflow Fix (D-07, MOB-01)

**Problem:** Horizontal scroll leaks outside AppLayout on mobile because fixed-width children (Kanban, tables) extend beyond `100vw`.

**Contract:**
- The outermost `<div>` in AppLayout (`flex min-h-screen`) must carry `overflow-x: hidden`. Add class `overflow-x-hidden` to this div.
- The inner flex `<div className="flex-1 flex flex-col min-w-0 overflow-hidden">` already has `overflow-hidden` — verify this persists.
- `<main>` already has `overflow-auto` — this must remain so Kanban's internal scroll works.
- `<body>` in `app.blade.php` does **not** need `overflow-x: hidden` — the AppLayout container fix is sufficient and avoids interfering with full-screen modal positioning.

**Acceptance:** No horizontal scrollbar visible on any page at 375px viewport width.

---

### 2. KanbanBoard — Mobile Scroll (D-01, D-02, MOB-02)

**Problem:** Kanban uses `overflow-x-auto kanban-scroll` on its column container. This scroll currently propagates to the page level.

**Contract:**
- KanbanBoard root `<div className="relative h-full w-full">` needs a containing context. Add `overflow-hidden` to it.
- The inner column strip `<div className="flex h-full gap-4 overflow-x-auto kanban-scroll pb-2">` keeps its `overflow-x-auto` — this is the intentional horizontal scroll zone (D-01).
- On mobile the Kanban container must have an explicit height so vertical overflow does not collapse. Use `h-[calc(100dvh-8rem)]` on mobile and `h-full` on desktop: `h-[calc(100dvh-8rem)] md:h-full`.
- Touch drag-and-drop: `TouchSensor` is **already configured** in `KanbanBoard.tsx` (line 177: `useSensor(TouchSensor, { activationConstraint: { delay: 200, tolerance: 8 } })`). D-02 is already done — no change needed. Verify only during testing.
- `kanban-scroll` custom scrollbar is hidden on mobile via `-webkit-scrollbar` display:none already in `.no-scrollbar` — but `kanban-scroll` has its own scrollbar style. On mobile the 6px scrollbar is fine — it aids discoverability of horizontal content.

**Acceptance:** Swipe to scroll between Kanban columns on mobile without triggering page scroll.

### 3. DemandDetailModal — Full-Screen Mobile (D-03, MOB-02)

**Current state:** Modal renders as `fixed inset-0 z-50 flex items-center justify-center p-4 md:p-6` with inner div `rounded-[16px] max-h-[90vh] w-full max-w-6xl`.

**Contract:**
- Outer wrapper keeps `fixed inset-0 z-50 flex items-center justify-center` but change padding: `p-0 md:p-6`.
- Inner modal panel: replace `rounded-[16px] max-h-[90vh] max-w-6xl` with responsive classes:
  - Mobile: `rounded-none max-h-[100dvh] max-w-full h-[100dvh]` (full bleed, no rounded corners)
  - Desktop: `md:rounded-[16px] md:max-h-[90vh] md:max-w-6xl md:h-auto`
  - Combined: `rounded-none md:rounded-[16px] h-[100dvh] md:h-auto max-h-[100dvh] md:max-h-[90vh] w-full md:max-w-6xl`
- The backdrop div (`absolute inset-0 bg-black/40 backdrop-blur-sm`) stays as-is.
- **Swipe down to close:** Add a visual drag handle bar at the very top of the modal on mobile only: a `4px × 32px` pill shape centered, color `#374151` (dark) / `#d1d5db` (light), with `8px` top margin. Hidden on desktop (`md:hidden`). This handle is purely visual — it signals swipe affordance. Actual swipe gesture implementation uses a `touchstart`/`touchmove` handler: if user swipes down more than 80px from top, call `onClose()`. On desktop, swipe gesture is not active.
- **Close button (X):** Already present at line 275 — keep as-is. On mobile the X button is the primary close action when the user does not swipe.
- **Body layout on mobile:** The two-column body grid (`grid grid-cols-1 gap-0 md:grid-cols-2`) already stacks to 1 column on mobile — no change needed.
- **Tab list on mobile:** Tabs (Arquivos, Comentários, Brief, Chat) remain as horizontal tab list. No conversion to dropdown. Ensure tab labels truncate gracefully on 375px — verify label width.

**Animation:**
- Entry: `translate-y-full` → `translate-y-0` over 250ms ease-out (slide up from bottom on mobile).
- Exit: `translate-y-0` → `translate-y-full` over 200ms ease-in.
- Implementation: CSS transition class toggle via `useState(open)` or `data-state` attribute pattern.
- On desktop: no slide animation — keep current behavior (instant render behind backdrop).
- Use Tailwind `transition-transform duration-250 ease-out` — do not use a third-party animation library.

**Swipe handle visual spec:**
```
position: absolute, top: 8px, left: 50%, transform: translateX(-50%)
width: 32px, height: 4px
background: #374151 (dark) / #d1d5db (light)
border-radius: 9999px
display: block on mobile (md:hidden)
```

### 4. Dashboard — Single Column (D-04, MOB-02)

**Current state:** Dashboard uses multiple responsive grid breakpoints.

| Grid | Current classes | Mobile result |
|------|----------------|---------------|
| Status cards | `grid-cols-2 sm:grid-cols-3 lg:grid-cols-5` | 2 columns — acceptable |
| Main 3-col grid | `grid-cols-1 md:grid-cols-3` | Already 1 column on mobile — no change needed |
| Charts 2-col grid | `grid-cols-1 lg:grid-cols-2` | Already 1 column on mobile — no change needed |

**Contract:**
- Status cards row (`grid-cols-2 sm:grid-cols-3 lg:grid-cols-5`) stays at 2 columns on mobile. This is correct — 2 cols fits 375px well with 4px gap. **No change needed.**
- The 3-column main grid (`grid-cols-1 md:grid-cols-3`) already collapses to 1 column on mobile. **No change needed.**
- Chart grids already collapse. **No change needed.**
- The `DashboardPlanningWidget` and `ActivityFeed` already stack vertically below `md:`. **No change needed.**
- Dashboard is already mostly responsive. Only verify that no individual card has a fixed pixel width that bleeds past 375px.

**Visual stacking order on mobile (top to bottom):**
1. Status cards (2-col grid)
2. "Seu foco agora" card
3. "Minhas demandas" card
4. "Bloqueadas / aguardando" card
5. Charts section
6. Activity feed

### 5. PWA Manifest + Icons (D-05, MOB-03)

**Current state (already exists):**
- `public/manifest.json` — present with correct fields
- `public/icons/icon-192.png` — present
- `public/icons/icon-512.png` — present
- `public/icons/apple-touch-icon.png` — present
- `<link rel="manifest" href="/manifest.json">` — already in `app.blade.php` (line 17)
- `<meta name="theme-color" content="#7c3aed">` — already in `app.blade.php` (line 10)
- Apple meta tags — already present (lines 11–13)
- `sw.js` — stub present (passthrough fetch handler, no caching)

**Contract:**
- Validate the manifest.json `icons` array: **the `purpose` field must be split** — combining `"any maskable"` in a single entry is non-standard. Correct to two separate entries:
  ```json
  { "src": "/icons/icon-192.png", "sizes": "192x192", "type": "image/png", "purpose": "any" },
  { "src": "/icons/icon-192.png", "sizes": "192x192", "type": "image/png", "purpose": "maskable" },
  { "src": "/icons/icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "any" },
  { "src": "/icons/icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
  ```
- Add `"id": "/dashboard"` to manifest.json (recommended by Chrome for installability — prevents duplicate install entries across different start_urls).
- Viewport meta tag: `<meta name="viewport" content="width=device-width, initial-scale=1">` is already present (line 5 of app.blade.php) — no change.
- Service worker registration script is already present in `app.blade.php` (lines 18–21). The current `sw.js` is a network-first passthrough — this matches D-06 (no offline cache). **No changes to sw.js this phase.**
- Icons must already exist on disk — they do. Verify icon files are actual image files (not placeholders) during testing.
- `display: standalone` is already set — this is what enables "Add to Home Screen" on iOS and Android.

**Acceptance criteria for MOB-03:** Chrome DevTools Lighthouse "Installable" audit must pass without errors.
