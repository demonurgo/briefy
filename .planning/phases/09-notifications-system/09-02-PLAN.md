<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Reverb WebSocket → AppLayout.tsx | Echo receives broadcast events on the org channel; payload is server-generated but arrives via WebSocket |
| AppLayout.tsx → user_id filter | Frontend must verify event.user_id matches auth.user.id before acting (D-07) |
| Frontend → POST /notifications/read-all | Authenticated user bulk-marks notifications as read |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-07 | Spoofing | AppLayout Echo event.user_id | mitigate | Frontend filters `event.user_id !== auth.user.id` before incrementing badge (D-07); channel authorization in channels.php prevents cross-org subscriptions |
| T-09-08 | Information Disclosure | Shared org channel | mitigate | Channel is `private-organization.{orgId}` — subscription requires auth; channels.php verifies `current_organization_id` matches; cross-org data exposure blocked at auth layer |
| T-09-09 | Tampering | Local unreadCount state | accept | Local state is UI-only display; actual notification read state is authoritative on server; opening bell dropdown re-fetches from server (D-09), correcting any desync |
| T-09-10 | Denial of Service | Rapid notification events → rapid badge increments | accept | Increments are O(1) local state updates; no DOM thrashing or server round-trips triggered; benign at v1.3 scale |
| T-09-11| Elevation of Privilege | POST /notifications/read-all | accept | Route is scoped to `auth()->id()` — marks only the current user's own notifications; no cross-user effect possible |
</threat_model>
